# 品質評価基準（4基準）

記事の品質を評価する4つの基準。**すべての基準を満たす必要があります。**

## 1. 希少性

**質問**: 巷の情報では得られない実践知か？

### 良い例

- **本番環境で実際に遭遇したトラブルと解決策**
  - 公式ドキュメントには書いていない落とし穴
  - 実運用で初めて分かった問題と対処法

- **Remix × Cloudflare Edgeの組み合わせ特有の知見**
  - 特定の組み合わせでのみ発生する問題
  - エコシステム固有の最適化手法

- **6年間のプロダクト運用経験から得られた「延命」の知恵**
  - 長期運用で得られた設計の知見
  - プロダクトの寿命を見抜く眼力

### 悪い例

- **公式ドキュメントの焼き直し**
  - 公式ドキュメントに書いてある内容の翻訳
  - チュートリアルの再現

- **既に多数の記事が存在する一般的なトピック**
  - 「Remixの始め方」のような基本的な内容
  - 既に多数の記事がある内容

- **単なる設定ファイルの古い記述の修正**
  - deprecatedな設定の修正のみ
  - バージョンアップに伴う単純な変更

### 判定基準

- **高い希少性**: 巷の情報では得られない、本番環境での実践知
- **低い希少性**: 公式ドキュメントや既存記事で容易に入手可能な情報

**判定方法**:
1. 類似記事を検索（Google, Zenn, Qiita等）
2. 公式ドキュメントに記載があるかを確認
3. 「巷の情報では得られない」と言えるかを判断

---

## 2. 具体性

**質問**: 読者が明日から試せる具体的なTipsか？

### 良い例

- **再現可能なコードスニペット**
  ```typescript
  export async function loader({ request }: LoaderArgs) {
    // 具体的なコード例
  }
  ```

- **具体的な設定ファイルの変更例**
  ```json
  {
    "compilerOptions": {
      "target": "ES2022"
    }
  }
  ```

- **ステップバイステップの実装手順**
  1. ファイルAを作成
  2. コマンドBを実行
  3. 設定Cを変更

### 悪い例

- **抽象的な概念の説明のみ**
  - 「アーキテクチャを改善すべき」（どうやって？）
  - 「パフォーマンスを最適化」（具体的には？）

- **コード例のない理論的な議論**
  - 概念の説明のみで、実装例がない
  - 「〜すると良い」という主張のみ

- **「〜すべき」という主張のみ**
  - 具体的な手順がない
  - 読者が実践できない

### 判定基準

- **高い具体性**: 読者が明日から試せる、再現可能な手順・コード例
- **低い具体性**: 抽象的で、実践方法が不明

**判定方法**:
1. コードスニペットが含まれているか
2. 具体的な設定ファイル・コマンドが示されているか
3. 「読者が明日から試せる」と言えるかを判断

---

## 3. 新規性

**質問**: 直近1年以内に登場した新しい要素を含むか？

### 良い例

- **Claude Code Projects（2024年登場）**
  - .claude/claude-config.json の活用
  - プロジェクト固有の設定

- **Slash Commands（2024年登場）**
  - カスタムコマンドの作成
  - スキルの活用

- **MCP（Model Context Protocol）（2024年登場）**
  - MCPサーバーの作成
  - 外部ツールとの連携

- **Cloudflare D1の新機能**
  - 最新のD1機能
  - パフォーマンス改善

### 悪い例

- **数年前から存在する機能の説明**
  - Remixの基本的なLoader/Action（2021年〜）
  - React Hooksの基本（2019年〜）

- **既に広く知られている手法**
  - TypeScriptの型定義の基本
  - Tailwind CSSの基本的な使い方

- **レガシーな技術の紹介**
  - jQuery（2006年〜）
  - Backbone.js（2010年〜）

### 判定基準

- **高い新規性**: 直近1年以内に登場した要素を含む
- **低い新規性**: 数年前から存在する機能のみ

**判定方法**:
1. 登場時期を確認（公式ドキュメント、リリースノート）
2. 「直近1年以内」を厳格にチェック
3. 新しい要素が記事の中心になっているかを確認

**基準日**: 2025年1月（現在から1年前 = 2024年1月以降に登場）

---

## 4. 実証性

**質問**: 本番環境で検証済みの情報か？

### 良い例

- **このブログ自体で実践している内容**
  - ClaudeMixブログで実装済み
  - 実際に動作している機能

- **LighthouseスコアAll 100という測定可能な結果**
  - 具体的な数値で検証
  - 再現可能な結果

- **6年間の現場経験に基づく知見**
  - 長期運用での検証
  - 複数のプロダクトでの実践

### 悪い例

- **机上の空論**
  - 実装していない理論
  - 検証していない推測

- **検証していない推測**
  - 「〜だと思う」「〜かもしれない」
  - 根拠のない主張

- **他人の記事の引用のみ**
  - 自身の経験がない
  - 二次情報のみ

### 判定基準

- **高い実証性**: 本番環境で検証済み、測定可能な結果がある
- **低い実証性**: 机上の空論、検証していない推測

**判定方法**:
1. 本番環境で検証済みかを確認
2. 測定可能な結果があるかをチェック
3. 自身の経験に基づいているかを確認

---

## 品質評価のプロセス

### ステップ1: 各基準をチェック

4つの基準それぞれについて、記事の内容が基準を満たしているかをチェックしてください。

**チェック方法**:
- [ ] **希少性**: 巷の情報では得られない実践知か？
- [ ] **具体性**: 読者が明日から試せる具体的なTipsか？
- [ ] **新規性**: 直近1年以内に登場した新しい要素を含むか？
- [ ] **実証性**: 本番環境で検証済みの情報か？

### ステップ2: 満たしていない基準を特定

満たしていない基準がある場合、その理由を明確化してください。

**例**:
- **希少性**: 既に多数の記事が存在する一般的なトピック
- **新規性**: 数年前から存在する機能のみ

### ステップ3: 改善の選択肢を提示

満たしていない基準がある場合、改善の選択肢を提示してください。

**選択肢**:
1. **内容を追加する**: 希少性・新規性の高い要素を追加
2. **記事を見送る**: 品質基準を満たせない場合
3. **別の角度から書く**: Challenge/Solutionを変更

### ステップ4: 総合評価

すべての基準を満たしている場合は「合格」、そうでない場合は「要改善」としてください。

**総合評価**:
- **合格**: すべての基準を満たしている
- **要改善**: 1つ以上の基準を満たしていない

## 品質評価の例

### 例1: 合格

**記事**: 「Claude Code Projectsで自動テスト環境を構築する完全ガイド」

- [x] **希少性**: Claude Code Projectsの実践的な活用方法（公式ドキュメントにない）
- [x] **具体性**: 再現可能なコード例、ステップバイステップの手順
- [x] **新規性**: Claude Code Projects（2024年登場）
- [x] **実証性**: ClaudeMixブログで実装済み

**総合評価**: 合格

### 例2: 要改善

**記事**: 「Remixの始め方」

- [ ] **希少性**: 公式ドキュメントの焼き直し、既に多数の記事が存在
- [x] **具体性**: 再現可能なコード例がある
- [ ] **新規性**: Remixの基本的な機能のみ（2021年〜）
- [ ] **実証性**: 自身の経験に基づいていない

**総合評価**: 要改善

**改善の選択肢**:
1. **内容を追加する**: Claude Code × Remixの組み合わせ特有の知見を追加
2. **記事を見送る**: 既存記事と差別化できない
3. **別の角度から書く**: 「Remix × Cloudflare Edgeでのパフォーマンス最適化」に変更

## よくある質問

### Q1: すべての基準を満たす必要がありますか？

**A**: はい、すべての基準を満たす必要があります。1つでも満たしていない場合は「要改善」となります。

### Q2: 希少性が低い内容はどうすれば良いですか？

**A**: 希少性の高い内容に注力し、低い内容は簡潔に済ませてください。または、記事を見送ることも検討してください。

### Q3: 新規性の基準日はいつですか？

**A**: 現在から1年前です。2025年1月現在であれば、2024年1月以降に登場した要素が新規性ありと判定されます。

### Q4: 実証性は自身の経験でなければいけませんか？

**A**: はい、自身の経験に基づいた内容である必要があります。他人の記事の引用のみの場合は実証性なしと判定されます。
