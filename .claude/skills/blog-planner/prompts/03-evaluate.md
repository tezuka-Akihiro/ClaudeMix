# ステップ3: 提供価値評価プロンプト

## AI役割定義

あなたは記事品質評価の専門家です。
記事の提供価値を3つのカテゴリで分類し、4つの品質基準で評価してください。

## 前提条件

ステップ2で整理したWhy/What/Challenge/Solutionを基に、提供価値と品質を評価します。

**参照**:
- `docs/quality-criteria.md` - 4つの品質基準
- `docs/value-categories.md` - 提供価値の3カテゴリ

## 思考プロセス（CoT）

以下の順序で段階的に評価してください：

```text
Step 1: 提供価値の分類
  → 3つのカテゴリ（ClaudeMix ガイド / ClaudeMix 記録 / ClaudeMix 考察）のどれに該当するか？

Step 2: 品質基準の適用
  → 4つの基準（希少性 / 具体性 / 新規性 / 実証性）を満たしているか？

Step 3: 価値の低い内容の見直し
  → 基準を満たしていない場合、どう改善するか？
```

## 実行手順

### 1. 提供価値の分類

記事の提供価値を3つのカテゴリで分類してください。

**詳細**: `docs/value-categories.md`

| カテゴリ | 説明 | 例 |
| :--- | :--- | :--- |
| **ClaudeMix ガイド** | 公式ドキュメントの翻訳・まとめ | MCP, Skills, Prompts, Rules |
| **ClaudeMix 記録** | トラブルシュート・最適化ログ | Troubleshooting, Performance, E2E |
| **ClaudeMix 考察** | 設計思想・リファクタリング知見 | Architecture, Refactoring, Type Design |

**ポイント**:
- 複数のカテゴリに該当する場合、すべて記載
- 該当なしの場合は「なし」と記載

### 2. 品質基準の適用

記事の品質を4つの基準で評価してください。**すべての基準を満たす必要があります。**

**詳細**: `docs/quality-criteria.md`

#### ① 希少性

**質問**: 巷の情報では得られない実践知か？

**良い例**:
- 本番環境で実際に遭遇したトラブルと解決策
- Remix × Cloudflare Edgeの組み合わせ特有の知見
- 6年間のプロダクト運用経験から得られた「延命」の知恵

**悪い例**:
- 公式ドキュメントの焼き直し
- 既に多数の記事が存在する一般的なトピック
- 単なる設定ファイルの古い記述の修正

#### ② 具体性

**質問**: 読者が明日から試せる具体的なTipsか？

**良い例**:
- 再現可能なコードスニペット
- 具体的な設定ファイルの変更例
- ステップバイステップの実装手順

**悪い例**:
- 抽象的な概念の説明のみ
- コード例のない理論的な議論
- 「〜すべき」という主張のみ

#### ③ 新規性

**質問**: 直近1年以内に登場した新しい要素を含むか？

**良い例**:
- Claude Code Projects（2024年登場）
- Slash Commands（2024年登場）
- MCP（Model Context Protocol）（2024年登場）
- Cloudflare D1の新機能

**悪い例**:
- 数年前から存在する機能の説明
- 既に広く知られている手法
- レガシーな技術の紹介

#### ④ 実証性

**質問**: 本番環境で検証済みの情報か？

**良い例**:
- このブログ自体で実践している内容
- LighthouseスコアAll 100という測定可能な結果
- 6年間の現場経験に基づく知見

**悪い例**:
- 机上の空論
- 検証していない推測
- 他人の記事の引用のみ

### 3. 価値の低い内容の見直し

品質評価で「要改善」となった場合、以下のガイドラインに従って内容を見直してください。

**基本方針**:
- **希少性が低い内容は詳述しない**: 既存の情報は簡潔に、希少性の高い内容に注力
- **新規性のある内容を優先**: 直近1年以内の新しい要素を中心に構成

**改善の選択肢**:

1. **内容を追加する**: 希少性・新規性の高い要素を追加
2. **記事を見送る**: 品質基準を満たせない場合
3. **別の角度から書く**: Challenge/Solutionを変更

## 完了条件チェックリスト

- [ ] 提供価値を3つのカテゴリで分類した
- [ ] 4つの品質基準を適用した
- [ ] すべての基準を満たしているか確認した
- [ ] 満たしていない場合、改善の選択肢を提示した

## 評価のベストプラクティス

### ✅ 良い評価

**提供価値の分類**:
- 複数のカテゴリに該当する場合、すべて記載
- 各カテゴリの提供価値を具体的に記述

**品質基準の適用**:
- 各基準について、具体的な理由を記述
- すべての基準を満たしているかを厳格にチェック

**改善の選択肢**:
- 具体的な改善策を提示
- オペレーターが判断しやすい選択肢を提供

### ❌ 悪い評価

**提供価値の分類**:
- 抽象的で具体性がない
- 該当するカテゴリを見逃している

**品質基準の適用**:
- 理由が抽象的で具体性がない
- 基準を満たしていないのにチェックしている

**改善の選択肢**:
- 抽象的で実行できない
- 選択肢が1つしかない

## 評価のポイント

### 希少性の評価

**高い希少性**:
- 本番環境での実践知
- 組み合わせ特有の知見
- 長期運用経験から得られた知見

**低い希少性**:
- 公式ドキュメントの焼き直し
- 既に多数の記事が存在するトピック

**判定**:
- 類似記事を検索して確認（必要に応じて）
- 「巷の情報では得られない」かを厳格にチェック

### 具体性の評価

**高い具体性**:
- 再現可能なコードスニペット
- 具体的な設定ファイルの変更例
- ステップバイステップの実装手順

**低い具体性**:
- 抽象的な概念の説明のみ
- コード例のない理論的な議論

**判定**:
- 「読者が明日から試せる」かを確認
- コード例、設定例、手順が含まれているかをチェック

### 新規性の評価

**高い新規性**:
- 直近1年以内に登場した要素
- Claude Code Projects（2024年）
- MCP（2024年）

**低い新規性**:
- 数年前から存在する機能
- 既に広く知られている手法

**判定**:
- 「直近1年以内」を厳格にチェック
- 新しい要素が中心になっているかを確認

### 実証性の評価

**高い実証性**:
- このブログ自体で実践している内容
- 測定可能な結果（Lighthouseスコア等）
- 本番環境での検証

**低い実証性**:
- 机上の空論
- 検証していない推測

**判定**:
- 「本番環境で検証済み」かを確認
- 測定可能な結果があるかをチェック
