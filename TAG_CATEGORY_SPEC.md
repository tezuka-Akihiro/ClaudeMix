# ブログ タグ・カテゴリ仕様書

**最終更新日**: 2025-11-26

このドキュメントは、ブログ記事の分類に使用するカテゴリとタグの仕様を定義します。
すべての記事は、この仕様に基づいて分類してください。
定義のマスターデータは `develop/blog/posts/spec.yaml` にあります。

---

## 1. 基本原則

- **役割分担**: カテゴリで記事の「価値・位置づけ」を大別し、タグで具体的な「技術要素」や「性質」を示します。
- **唯一の情報源**: `spec.yaml` を唯一の信頼できる情報源（Single Source of Truth）とします。タグやカテゴリの追加・変更は `spec.yaml` で行ってください。
- **シンプルさ**: タグは階層構造を持たないフラットなリストとして管理します。

---

## 2. カテゴリ仕様

カテゴリは以下の3つに固定します。記事の目的や読者に提供する価値に応じて、1つの記事に1つのカテゴリを割り当ててください。

| ID | カテゴリ名 | Emoji | 説明 | 主な用途 |
| :-- | :--- | :--- | :--- | :--- |
| 1 | **Claude Best Practices** | 📚 | 実装やトラブルシュートから得られた実践的な知見やノウハウを共有します。 | トラブルシュート、技術的なベストプラクティスなど（記事全体の約60-70%） |
| 2 | **ClaudeMix Philosophy** | 🎨 | 設計思想、アーキテクチャ、リファクタリングなど、プロジェクトの根幹に関わる考えをまとめます。 | 月1回の「学んだことまとめ」など（記事全体の約5-10%） |
| 3 | **Tutorials & Use Cases** | 🚀 | 新機能の追加や特定機能の使い方など、具体的な手順を解説します。 | ハンズオン形式のチュートリアルなど（記事全体の約20-30%） |

### 注意事項
- **「Troubleshooting」カテゴリは存在しません。**
  - トラブルシュートを主題とする記事は、そこから得られた知見をまとめる形で `Claude Best Practices` カテゴリに分類し、性質タグとして `troubleshooting` を付与してください。

---

## 3. タグ仕様

タグは、記事で扱っている具体的な技術や記事の性質を示すために使用します。

### 3.1. タグの分類

タグは以下の2種類に大別されます。

| 種類 | 説明 | 例 |
| :--- | :--- | :--- |
| **技術タグ (technical)** | 記事で扱う具体的な技術、ライブラリ、ツール名など。 | `SSR`, `Vite`, `Playwright`, `MCP` |
| **性質タグ (nature)** | 記事が書かれた背景や種類を示します。 | `troubleshooting`, `refactoring`, `testing` |

### 3.2. タグ一覧

記事作成時には、以下のリストを参考に適切なタグを選択してください。
YAGNI (You Ain't Gonna Need It) の原則に基づき、実際に記事で言及されている技術のみをタグとしてください。

#### 現在使用中のタグ

- **Remix関連**: `SSR`, `Vite`, `React`
- **Cloudflare関連**: `Workers`
- **Claude Code関連**: `MCP`, `Skills`, `Prompts`, `Projects`
- **性質**: `troubleshooting`, `refactoring`, `testing`, `architecture`

#### 推奨タグ（必要に応じて使用）

- **Remix関連**: `Loader`, `Action`, `Vitest`, `Playwright`
- **Cloudflare関連**: `Pages`, `KV`, `D1`
- **性質**: `UX`, `performance`

### 3.3. 命名規則と運用ルール

- **自明なタグは不要**: プロジェクトの核となる技術（例: `Remix`, `Cloudflare`）は、全ての記事の前提となるためタグとして付与しません。
- **フォーマット**:
  - 原則として小文字を使用します。（例: `testing`）
  - 固有名詞や広く認知された略語は大文字を許可します。（例: `SSR`, `MCP`）
  - 複合語はハイフンでつなぎます。（例: `best-practices`）
- **新規タグの追加**:
  1. `spec.yaml` の `tags` セクションに、類似タグがないか確認します。
  2. 命名規則に従っているか確認します。
  3. `spec.yaml` にタグを追加し、その定義をコミットに含めてから記事で使用します。

## 4. タグ運用フロー

タグの選定や新規追加の運用は、AIエージェントのプロンプトで厳密に定義されています。
**開発者が手動でタグを追加・変更することは原則として禁止です。**

タグ運用に関する詳細なフローや基準については、以下のAIエージェント向けプロンプト定義を参照してください。

- **参照先**: `.claude/prompts/blog-article-writer.md`

## 5. 検索シナリオ例

この仕様により、ユーザーは以下のような柔軟な検索が可能になります。

- **シナリオ1**: Playwrightに関するトラブルシュート記事を探したい
  - **`Playwright`** (技術タグ) と **`troubleshooting`** (性質タグ) で絞り込む。

- **シナリオ2**: 設計思想に関する記事をまとめて読みたい
  - **`ClaudeMix Philosophy`** (カテゴリ) を選択する。

- **シナリオ3**: SSRに関するチュートリアルを探したい
  - **`Tutorials & Use Cases`** (カテゴリ) と **`SSR`** (技術タグ) で絞り込む。
