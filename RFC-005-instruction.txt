📌 タスク概要
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

YAML参照機能（anchor/alias、merge key）をサポートし、specファイル間での値の再利用を
可能にする。これにより、spec定義の柔軟性が向上し、さらなるDRY原則の徹底が実現する。

重要: js-yamlは既にanchor/alias/merge keyをサポートしているため、新規実装は不要。
      このPhaseは主に「使用ガイドの整備」と「パイロット実装」が中心。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎯 目的
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. YAML anchor/aliasのサポート: spec内での値の再利用
2. YAML merge keyのサポート: デフォルト値の継承
3. 将来の拡張準備: JSONポインタ形式の $ref サポート（Phase 6以降）

【Phase 1: 準備と既存機能確認】

1. RFC-005ドキュメントを読んで全体像を把握

2. js-yamlの既存サポートを確認（RFC-005の186-227行参照）:

   テストコード例:

   const yaml = require('js-yaml');

   const yamlString = `
   defaults: &defaults
     timeout: 5000
     retry: 3

   loader:
     <<: *defaults
     method: "GET"
   `;

   const parsed = yaml.load(yamlString);
   console.log(parsed);
   // anchor/aliasが正しく展開されることを確認

3. 新規ブランチ作成: git checkout -b claude/rfc-005-yaml-ref-XXXXX

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【Phase 2: 使用ガイドドキュメント作成】

1. docs/specs/YAML_REFERENCE_GUIDE.md を作成
   （RFC-005の469-647行参照）

主要セクション:
- 概要
- 基本的な使い方（Anchor と Alias）
- Merge Keyの使い方（継承と上書き）
- 実践例
  - 例1: フォームフィールドの共通化
  - 例2: UIセレクタのパターン化
- ベストプラクティス（DO/DON'T）
- トラブルシューティング
  - "unknown anchor" エラー
  - TypeScriptエラー "_templates does not exist"
- 制限事項
  - spec間の参照は不可（同一ファイル内のみ）
  - JSONポインタ形式は未サポート
  - 動的な値生成は不可

【Phase 3: パイロット実装】

1. account/authentication-spec.yaml でanchor/aliasを試験的に使用
   （RFC-005の142-177行参照）

例:
_field_defaults: &field_defaults
  required: true

_email_field: &email_field
  <<: *field_defaults
  label: "メールアドレス"
  placeholder: "example@example.com"
  type: "email"
  autocomplete: "email"

_password_field: &password_field
  <<: *field_defaults
  label: "パスワード"
  placeholder: "8文字以上"
  type: "password"

forms:
  login:
    email:
      <<: *email_field
      autocomplete: "email"
    password:
      <<: *password_field
      autocomplete: "current-password"

  register:
    email:
      <<: *email_field
    password:
      <<: *password_field
      autocomplete: "new-password"

2. 命名規則の遵守（RFC-005の231-259行参照）:
   - anchor名はアンダースコア接頭辞（_defaults）
   - _templates セクションにanchor定義を集約
   - 意味が明確な名前を使用

【Phase 4: 型定義の更新】

1. account/types.ts を確認（RFC-005の261-284行参照）

重要: TypeScript型定義では _templates を除外

export interface AccountAuthenticationSpec {
  metadata: {...};
  // _templates は含めない（実行時には存在するが、型には不要）
  forms: {
    login: {...};
    register: {...};
  };
}

【Phase 5: 単体テスト作成】

1. scripts/prebuild/generate-specs.test.js にテスト追加
   （RFC-005の387-449行参照）

- 上書きが正しく動作する

例:
describe('YAML anchor/alias support', () => {
  it('anchorとaliasを正しく展開できる', () => {
    const yamlString = `
defaults: &defaults
  timeout: 5000

loader:
  <<: *defaults
  method: GET
`;

    const parsed = yaml.load(yamlString);

    expect(parsed.loader.timeout).toBe(5000);
    expect(parsed.loader.method).toBe('GET');
  });

  // 他のテストケースも実装
});
